# Execution Contract

## Supported route

The only authorized route is the official Codex CLI using an existing ChatGPT subscription session and its built-in `imagegen` skill. The helper does not force an agent or image model; Codex selects the supported image route. Reasoning effort remains `medium`.

No part of this skill may read authentication files, extract cookies, automate a web page, call a private endpoint, or use an API key. Authentication failures are terminal and must be reported without raw subprocess output.

## Model-label honesty

The request summary separates these fields:

- `requested_model`: `null` because the helper deliberately leaves model selection to Codex imagegen;
- `observed_model`: model identity returned as supported evidence, otherwise `null`;
- `model_identity_attested`: `false` unless supported evidence establishes identity.

A successful file does not by itself prove exact model identity. Say “generated through the official Codex CLI ChatGPT-subscription path.” Do not say “generated by GPT Image 2” unless the supported response provides evidence for that exact identity.

## Approval boundary

Dry-run is always allowed and is the default. Before the first live image, Vision inspection, or Telegram delivery, stop and request approval for that immediate action. Live generation additionally requires `--execute` and `HERMES_IMAGE_LIVE_APPROVED=1` on that single invocation.

Do not persist the marker in shell profiles, config, `.env`, or scripts. Remove it from the next invocation. A failure does not authorize a fallback or automatic retry.

## Output and delivery

The helper refuses collisions and missing output directories. It accepts zero to four existing reference files and does not mutate them.

For Telegram delivery, use Hermes `send_message` with the generated image as a document/file attachment and retain its filename. A successful Telegram API response is delivery evidence; printing the path is not.
